# Calendar-Date Anchor — `termSession` Replaced by `sessionType` + `holiday`

The preschool "session" entity is stored as `termSession`: one row per date + `start/end` + `label` + an `is_holiday` flag. This over-persists in three ways. (1) The time and label repeat on every row — if Session 1's time ever changes (planned: move to 30 min earlier next year; add Session 2), every past row must be rewritten or silently drifts to the new time. (2) Weekends and closures are persisted as rows with `is_holiday=true`; i.e. the _absence_ of a class is stored as presence in the table. (3) The date a capture refers to lives only by joining `termSession`; "everything that happened on 12 Feb" is an indirection through a row that exists for no other reason than to hold the date.

We are also mid-refactor toward a "calendar as anchor" model: any scheduled activity, capture, or audit lookup pivots off a date, not a generated session row. And two unrelated concepts are muddled under the word "session" — the auth `session` table (better-auth, tokens) and the preschool `termSession` — which confuses every reader.

**Decision — replace `termSession` with two single-purpose tables, anchored on date:**

1. **`sessionType` (the recurring time block).** `id, name, start, end, active`. Versioned by deactivate-and-insert: changing Session 1 from 9:00 to 8:30 next year = deactivate the old row, insert an active new one. Past captures keep their original times because they point at the deactivated row. Constraints: `unique(name, start, end)` for row identity + a partial `unique(name) where active = true` so only one slot per name is live. Planning and capture UIs filter `active = true` exclusively.

2. **`holiday` (closures, as a ranged event log).** `id, termId? (null, FK→term), startDate, endDate, reason, source ('manual' | 'synced'), scope ('national' | 'custom' | 'term')`. `termId = null` for global/national holidays (synced from the Indonesian holiday calendar or owner-entered), applicable to whatever term's range covers them; `termId = <value>` for term-local closures. Closures are ranged — Eid al-Fitr is one row spanning 5 days, not five. Multiple events may share a date (a national holiday _and_ a field trip on the same day), each its own deleteable row. Constraint `unique(termId, startDate, endDate, reason, source)` with NULLS DISTINCT prevents exact duplicates.

3. **Capture tables re-keyed off date**, not `termSession.id`:
   - `dailyClassReport`, `scheduleItem`, `reminderLog`: `unique(date, sessionTypeId)` — one class-level record per block. With only Session 1 active, one per day; Session 2 activating next year adds a second, no re-key.
   - `observation`, `dailyReportSnapshot`: `unique(kidId, date)` + `sessionTypeId` as an attribute — a kid attends exactly one session per day, so one observation/report per kid per day, tagged with which block they attended.

4. **School day derived, not stored.** A date is a school day iff it is a weekday, within the term's range, not covered by any `holiday` range, and at least one `sessionType` is active. Weekends are never persisted. "Is 12 Feb a holiday?" = `EXISTS holiday WHERE date BETWEEN startDate AND endDate`. "Everything on 12 Feb" = join holiday-overlap + captures-by-date + schedules-by-date + distinct active kids — one date-anchored read set, no `termSession` hop.

5. **Calendar UI.** reui.io's Calendar (built on the shadcn/ui Calendar / react-day-picker primitive — compliant with AGENTS.md rule #2) with its event-list-on-click and range-selection modes: one component covers date navigation, ranged-closure entry, and per-day event surfacing. Greyed-past overlay is a CSS modifier on cells `< today`, no data. reui is a custom shadcn registry (copy into source, not an npm dep) — its files live under `src/components/sections/` or a `src/components/reui/` subdir, never in the auto-generated `src/components/ui/`.

**Verification.** This is a data migration on the confirmation-gate list (DB schema migration). The migration backfills: each distinct `(name, start, end)` group from existing `termSession` rows becomes one `sessionType` row; rows with `is_holiday=true` become `holiday` events carrying their `date` and reason. Existing `dailyClassReport` / `observation` / `dailyReportSnapshot` rows carry only `sessionId` — each must be resolved to `(date, sessionTypeId)` via the old `termSession` row before the FK is dropped. Reversible only via backup. The rewrite does not land until an explicit go in a dedicated session; until then the codebase stays on `termSession` and this ADR records intent. `bun typecheck` + `bun lint` + a backfill-upsert dry-run on a snapshot DB are the verification gates.

**Tradeoffs.** We trade `termSession`'s single-table simplicity for two tables plus a partial-unique and a NULLS-DISTINCT constraint — more moving parts. Justified by three real wins: time-config changes stop rewriting history; closures stop duplicating per day (Eid is 1 row, not 5); and the date anchor makes both audit lookups ("what happened on 12 Feb") and the calendar UI direct queries rather than joins through a synthetic row. The partial unique and the ranged-overlap "is closed" check are the two non-trivial bits of machinery — both are standard Postgres and bounded. We accept that "a school day" is no longer a single-row lookup but a four-condition derivation; that's the right shape for a derived concept, and the alternative (persisting every school day as a row) is exactly the over-persistence we're removing. The per-kid `unique(kidId, date)` rule encodes the domain constraint "a kid attends one session per day" at the DB level rather than trusting app logic.

**Out of scope.** The holiday-sync source (Indonesian holiday API vs. manual entry) and the merge/duplicate-event UI are deferred — the table shape supports both; the exact sync mechanism and UI are later decisions. Promotion of a "calendar-date anchor" hard rule into AGENTS.md is held until the migration lands: AGENTS.md's own rule is "the code wins for now," so the hard rule lands _with_ the code, not before. This ADR is the correct home for the decision in the meantime.

**Cross-reference — collision with the in-flight feature refactor.** This ADR contradicts the active `src/lib/actions/` → `src/features/<entity>/` migration tracked in `docs/patterns.md`. The recently-migrated `src/features/session/` module (commits `f4be495`, `6e88d4d`) wraps exactly the `termSession` entity this ADR deletes. No conflict today — the feature module keeps working on the current schema — but at migration time the `session` feature will fold into a `session-type` + `holiday` restructure (one feature per new table), and the date-anchored capture re-key landing in `dailyClassReport`/`observation`/`dailyReportSnapshot` will follow the same feature shape. Reconcile the feature layout _with_ the migration, not as a separate pass, to avoid migrating `features/session/` onto a table that no longer exists.
